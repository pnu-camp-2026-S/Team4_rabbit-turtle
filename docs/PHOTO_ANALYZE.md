# Photo Taste Analyzer

## 목적

이 스킬은 사용자 사진을 추천에 사용할 수 있는 취향 신호로 바꾸기 위해 사용한다. AI 분석 결과는 최종 정답이 아니라, 사용자가 확인하고 고칠 수 있는 초안으로 다룬다.

목표는 사진 속 모든 물체를 나열하는 것이 아니다. 매거진이나 콘텐츠 추천 품질을 높일 수 있는 시각적, 맥락적 신호만 추출한다.

앱 화면에 노출되는 키워드는 반드시 `references/ui_keyword_vocabulary.md`의 UI 키워드 중에서만 고른다. 사진에 더 구체적인 대상이 보이더라도 가장 가까운 UI 키워드로 매핑하고, UI 목록에 없는 표현은 `keywords`와 `more_signals`에 넣지 않는다.

## 기본 흐름

1. 보이는 후보를 찾는다: 주요 객체, 배경/장소 유형, 활동 단서, 분위기, 가능한 장면 맥락을 확인한다.
2. 노이즈를 거른다: 우연히 찍힌 요소, 너무 작거나 부수적인 요소, 사생활에 민감한 요소, 추천과 무관한 요소를 제거한다.
3. 조합을 해석한다: 개별 객체 라벨보다 경험 단위의 의미를 우선한다.
4. 불확실성을 표시한다: 신뢰도를 함께 쓰고, 신원/정확한 위치/집/경제 수준/건강/종교/정치/관계에 대한 단정은 피한다.
5. 키워드 초안을 만든다: `references/ui_keyword_vocabulary.md` 안의 UI 키워드 중 가장 가까운 항목만 선택하고, 내부 매칭용 taxonomy `concept_id`를 함께 붙인다.
6. 사용자 피드백을 반영한다: 삭제/추가/수정된 키워드와 줄글 피드백을 구조화된 프로필 변경으로 변환한다.
7. 신중하게 승격한다: 한 장의 사진 결과는 후보로 저장하고, 사용자 확인 또는 반복 증거가 있을 때만 지속 취향으로 승격한다.

## 관련성 필터

시각 요소는 아래 질문 중 하나 이상에 답할 때만 남긴다.

- 사용자의 관심사를 설명하는가?
- 의도된 장소, 활동, 분위기, 콘텐츠 카테고리를 설명하는가?
- 여러 사진에서 반복될 경우 추천을 개선할 수 있는가?
- 우연한 배경이 아니라 사진의 의도적인 일부로 보이는가?

보통 남긴다:

- 장소 유형: 카페, 서점, 박물관, 갤러리, 문화재, 공원, 바다, 거리, 작업 공간.
- 활동: 산책, 독서, 커피, 식사, 방문, 휴식, 공부, 쇼핑, 촬영.
- 분위기/스타일: 조용한, 아늑한, 전통적인, 도시적인, 야외적인, 미니멀한, 활기찬, 경치 좋은.
- 콘텐츠 관심사: 커피, 디저트, 건축, 역사, 예술, 자연, 디자인, 로컬 탐방.

보통 건너뛴다:

- 우연히 찍힌 행인, 얼굴, 차량, 잡동사니, 작은 배경 물체, 부수적인 간판.
- 컵 모양이나 테이블 얼룩처럼 추천에 영향을 주기 어려운 과도한 세부 요소.
- 경제 수준, 건강, 종교, 정치, 정체성, 관계처럼 민감하거나 추측적인 속성.

## 맥락 해석

포커스를 정하기 전에 조합을 먼저 분석한다.

예: 문화재 배경 앞의 커피 사진은 `커피`만으로도, `문화재`만으로도 끝내지 않는다. UI 키워드 목록 안에서 다음과 같은 신호를 우선한다.

- `커피`
- `카페`
- `한옥`
- `골목 탐방`
- `조용한 휴식`

여러 해석이 가능하면 확실한 키워드와 불확실한 키워드를 분리한다. 위 예시에서는 `커피`, `전통 공간`은 높은 신뢰도로 둘 수 있지만, `여행`은 더 강한 단서가 없으면 낮은 신뢰도의 후보로만 둔다.

## 위치와 일상-여행 판단

사진만으로 정확한 위치를 추론하지 않는다. 명시적인 텍스트, 랜드마크 근거, 메타데이터, 사용자 입력이 있을 때만 구체적인 위치를 다룬다. GeoCLIP 같은 시각적 위치 추론 모델을 쓰더라도 넓은 지역감이나 장소 성격을 보완하는 약한 신호로만 사용한다.

선호하는 표현:

- `한국 전통 건축`
- `도심 문화재 공간`
- `관광지 같은 맥락`
- `일상적인 카페 방문 가능성`
- `비일상적인 문화 외출 가능성`

피해야 하는 표현:

- 정확한 좌표.
- 사용자의 집 근처라는 주장.
- 사용자가 여행 중이라는 단정.
- 거주지, 소득, 정체성, 개인 관계에 대한 추측.

일상인지 여행인지는 확률과 상태로 표현한다. 앱 기능에는 확인 질문이 없으므로 질문을 만들지 않는다.

```json
{
  "context_candidates": [
    { "ui_keyword": "홈라이프", "category": "LIFESTYLE", "mapped_concepts": ["context.daily_leisure", "mood.relaxed"], "confidence": 0.48, "status": "uncertain" },
    { "ui_keyword": "복합문화공간", "category": "SPACE", "mapped_concepts": ["context.cultural_outing", "culture.local_culture"], "confidence": 0.76, "status": "draft" },
    { "ui_keyword": "도시 여행", "category": "TRAVEL", "mapped_concepts": ["context.travel", "travel.city_landmark"], "confidence": 0.42, "status": "uncertain" }
  ]
}
```

## UI 키워드 매핑

사용자 화면에서는 단순한 칩처럼 보여주더라도, 앱에 노출되는 키워드는 모두 `references/ui_keyword_vocabulary.md`의 UI 키워드여야 한다. 내부 추천 매칭은 각 UI 키워드에 연결된 `mapped_concepts`의 taxonomy `concept_id`를 사용한다.

예:

- 사진 표현 `한옥 카페`, `전통 건축`, `문화재 앞 커피`는 UI 키워드 `한옥`, `카페`, `커피` 중에서 고른다.
- 매거진 표현 `고즈넉한 한옥 카페`, `서울 전통 공간 산책`도 UI 키워드 `한옥`, `카페`, `골목 탐방`, `조용한 휴식` 중에서 고른다.
- 정확히 맞는 UI 키워드가 없으면 가장 가까운 UI 키워드를 고르거나, 가까운 항목이 없을 때는 버린다.
- `more_signals`에도 UI 키워드 목록 안의 키워드만 넣는다. UI 목록에 없는 원문 표현은 넣지 않는다.

## `keywords`와 `more_signals`

사진 분석 결과는 사용자가 바로 확인할 수 있는 두 묶음으로 나눈다.

- `keywords`: 사진의 중심 소재, 장소, 활동, 분위기를 설명하는 강한 후보. 사용자가 그대로 남길 가능성이 높은 키워드다.
- `more_signals`: "이런 것도 관심 있나요?"로 보여줄 보조 후보. 사진에 약하게 보이거나, 맥락상 가능하지만 중심 키워드로 확정하기 어려운 UI 키워드다.

선정 규칙:

- `keywords`와 `more_signals`는 모두 `references/ui_keyword_vocabulary.md`에 있는 UI 키워드만 사용한다.
- 같은 `ui_keyword`를 `keywords`와 `more_signals`에 동시에 넣지 않는다.
- `keywords`는 보통 2-5개, `more_signals`는 보통 0-4개로 제한한다.
- `more_signals`에는 신뢰도 낮은 추측을 많이 넣지 말고, 사용자가 선택하면 추천 품질이 실제로 달라질 만한 후보만 넣는다.
- UI 목록 밖 표현이 보이면 가까운 UI 키워드로 흡수한다. 가까운 키워드가 없으면 `more_signals`에도 넣지 않는다.
- 사람, 얼굴, 사생활 단서, 정확한 위치, 집 근처 여부, 경제 수준 같은 민감하거나 부정확한 추론은 `more_signals`에도 넣지 않는다.

예:

- 한옥 카페 사진: `keywords`는 `한옥`, `카페`, `커피`; `more_signals`는 `조용한 휴식`, `골목 탐방`.
- 바르셀로나 축구장 사진: `keywords`는 `축구`, `경기장 투어`; `more_signals`는 `스포츠 관람`, `스포츠 여행`, `랜드마크`.
- 재즈바 공연 사진: `keywords`는 `재즈`, `라이브 공연`; `more_signals`는 `바이닐`, `와인`.
- 베이킹 클래스 사진: `keywords`는 `베이커리`, `디저트`; `more_signals`는 `취미 수집`, `카페`.

## 키워드 타입

표준 concept에 매핑할 때는 키워드 타입을 함께 유지한다.

사용할 카테고리:

- `object`: 커피, 책처럼 의미 있는 시각 객체.
- `place_type`: 카페, 박물관, 문화재, 공원, 서점.
- `activity`: 산책, 독서, 방문, 휴식, 커피 브레이크.
- `mood`: 조용한, 아늑한, 활기찬, 경치 좋은, 전통적인, 여유로운.
- `interest`: 역사, 건축, 커피, 예술, 자연, 로컬 문화.
- `context`: 일상 여가, 문화 외출, 여행 같은 맥락, 작업/공부.
- `preference`: 조용한 공간 선호, 문화+카페 선호처럼 지속 취향 후보.
- `negative_signal`: 사용자가 “아님”으로 고친 부정 신호.

화면 노출용 키워드 목록과 concept 매핑은 `references/ui_keyword_vocabulary.md`를 우선 참고한다. taxonomy 세부 구조가 필요할 때만 `references/taxonomy.md`를 참고한다.

## 출력 형식

프롬프트, API, 기능 로직을 설계할 때는 아래와 비슷한 출력을 사용한다.

```json
{
  "summary": "한옥과 카페 경험이 함께 보이는 사진입니다.",
  "keywords": [
    {
      "ui_keyword": "한옥",
      "category": "SPACE",
      "mapped_concepts": ["place.traditional_space", "culture.history_tradition"],
      "confidence": 0.93,
      "evidence": "배경의 전통 건축 요소",
      "status": "draft"
    },
    {
      "ui_keyword": "카페",
      "category": "FOOD",
      "mapped_concepts": ["place.cafe", "food_drink.coffee"],
      "confidence": 0.86,
      "evidence": "카페 공간 또는 음료가 중심 소재로 보임",
      "status": "draft"
    },
    {
      "ui_keyword": "커피",
      "category": "FOOD",
      "mapped_concepts": ["food_drink.coffee"],
      "confidence": 0.41,
      "evidence": "사진 속 음료가 커피로 보임",
      "status": "uncertain"
    }
  ],
  "more_signals": ["조용한 휴식", "골목 탐방"],
  "privacy_notes": ["정확한 위치와 집 근처 여부는 추정하지 않음"]
}
```

처음 사용자에게 보여주는 키워드는 적게 유지한다.

- 대표 키워드 3개.
- 보조 키워드 3-5개.
- 확인이 꼭 필요할 때만 불확실한 키워드 1-2개.

## 사용자 피드백 처리

다음 네 가지 사용자 행동을 지원한다.

- 키워드 삭제: 해당 사진 맥락에서 부정 신호 또는 가중치 하향 신호로 저장한다.
- 키워드 추가: 사용자가 직접 제공한 높은 신뢰도의 신호로 저장한다.
- 키워드 수정: AI 라벨을 사용자 라벨로 바꾸고 출처를 남긴다.
- 줄글 피드백 추가: 추가/삭제/가중치 하향/포커스 변경으로 구조화한다.

예시 줄글:

`여행은 아니고 집 근처야. 문화재보다는 조용한 분위기가 좋아서 찍었어.`

구조화 결과:

```json
{
  "remove": ["여행"],
  "add": ["동네 산책", "조용한 공간", "일상 속 휴식"],
  "downweight": ["문화재 중심"],
  "preference_focus": ["분위기", "휴식"],
  "negative_signal": ["여행 아님"]
}
```

## 프로필 업데이트 규칙

저장 계층을 세 가지로 분리한다.

- `photo_tags`: 한 장의 사진에서 AI가 관찰한 태그.
- `confirmed_tags`: 사용자가 확인, 수정, 추가한 태그.
- `preference_profile`: 여러 사진과 피드백을 통해 누적된 지속 취향.

아래 조건 중 하나가 없으면 한 번 나온 AI 관찰값을 곧바로 지속 취향으로 승격하지 않는다.

- 사용자가 명시적으로 확인했다.
- 여러 사진에서 같은 신호가 반복됐다.
- 사용자가 줄글로 그것을 취향이라고 밝혔다.

권장 승격 규칙:

```json
{
  "candidate": "조용한 공간",
  "promotion_reason": "user_confirmed_or_repeated",
  "profile_weight": 0.72,
  "sources": ["photo_feedback", "repeated_photo_signal"]
}
```

## 제품 UX 가이드

AI 결과는 권위 있는 분석이 아니라 수정 가능한 제안으로 보여준다.

좋은 문구:

- `사진에서 이런 관심사 후보를 찾았어요. 맞는 것만 남겨주세요.`
- `확실하지 않은 키워드는 확인이 필요해요.`

피해야 할 문구:

- `당신은 여행을 좋아합니다.`
- `이 장소는 집 근처입니다.`
- `당신의 취향 분석이 완료되었습니다.`

포커스가 불명확하면 질문을 만들지 말고 낮은 신뢰도 또는 `status: "uncertain"`으로 표시한다. 보조 신호가 필요해도 `more_signals`에는 `references/ui_keyword_vocabulary.md`에 있는 UI 키워드만 넣는다.

## 안전 경계

사용자가 명시적으로 제공했고 기능에 꼭 필요한 경우가 아니라면, 사진에서 민감한 개인정보를 추론하거나 저장하지 않는다.

피한다:

- 얼굴 신원, 나이, 성별, 민족, 건강, 종교, 정치, 소득, 관계 상태.
- 정확한 집/직장 위치나 장면이 집 근처인지 여부.
- `사용자는 과시를 중시한다`, `사용자는 외롭다` 같은 숨은 의도 단정.

불확실성 표현을 선호한다.

- `가능성`
- `후보`
- `맥락`
- `불확실`

## 구현 메모

모델 프롬프트에는 다음을 지시한다.

- 관련 없는 배경 잡음은 무시한다.
- `references/ui_keyword_vocabulary.md`의 UI 키워드 목록 안에서 가장 가까운 키워드만 선택한다.
- UI 키워드에 연결된 `mapped_concepts`를 함께 출력한다.
- 타입 대신 UI `category`, 신뢰도, 짧은 근거가 있는 키워드를 출력한다.
- 강한 후보는 `keywords`에, 약하지만 사용자가 선택할 만한 보조 후보는 `more_signals`에 넣는다.
- `more_signals`는 UI 키워드 문자열 배열로 출력하고, `keywords`와 중복시키지 않는다.
- 불확실한 맥락은 낮은 `confidence`와 `status: "uncertain"`으로 표시한다.
- 사용자에게 보여줄 보정 질문은 만들지 않는다.
- 사용자 피드백을 프로필 업데이트 작업으로 변환한다.

추천 시스템에서는 원본 AI 관찰값보다 사용자가 확인한 신호와 반복된 신호를 더 강하게 사용한다. 사용자 사진과 매거진 콘텐츠는 같은 UI 키워드와 그에 연결된 `mapped_concepts`를 기준으로 매칭한다.

## 매거진 분석에도 동일 적용

매거진 제목, 본문, 이미지 설명, 에디터 태그를 분석할 때도 사진 분석과 같은 UI 키워드 목록만 사용한다.

- 매거진에서 `재즈바`, `색소폰 공연`, `LP 바`가 나오면 UI 키워드 `재즈`, `라이브 공연`, `바이닐` 중에서 고른다.
- 매거진에서 `캄프 누`, `FC 바르셀로나`, `축구장 투어`가 나오면 UI 키워드 `축구`, `경기장 투어`, `스포츠 여행` 중에서 고른다.
- 매거진에서 목록 밖의 세부 표현이 나오더라도 UI에는 원문을 노출하지 않는다.
- `more_signals`에도 목록 안 키워드만 넣고, 목록 밖 표현은 내부 로그나 원문 근거로만 보관한다.
